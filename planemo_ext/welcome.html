Loading the ToolFactory sample history
  1. Use this URL: https://zenodo.org/record/4686436/files/TF_demo_planemo_april14.tar.gz?download=1
  2. This older one may be helpful if the above fails: https://zenodo.org/record/4661849/files/TFdemo_history_april_4.tar.gz?download=1
  3. Select the "User" tab of the top bar; Select Histories; Select Import; Paste that URL into the History import upload box
  4. Switch to the imported history for sample ToolFactory jobs to illustrate some of the kinds of tools and features.
Once the sample history is loaded, try re-running a tool job on any of the toolshed tarball history objects, using the circular arrow "Rerun" button. The ToolFactory form that generated the tool will appear, filled in the way it was when the sample was generated. Change the wording of prompts to see how they change. Experimenting with the examples may help you figure out how to use it for your work.
The Hello World example is a useful base model. Any working bash script with a few parameters can be substituted starting with that example as a model. Add parameters and outputs/inputs as needed.
Generated tools are just ordinary Galaxy tools. They work for any user and in workflows.
Generating a tool takes 1.5-4 minutes on my old desktop. Planemo is run twice and a Galaxy is built and torn down each time. The first run is to generate test outputs that are added to the new tool archive. The second time those sample outputs are tested. Conda adds time to install the dependencies and then there's the actual runtime for the test data
There should be a new history containing the demonstration tools. Each Toolshed tarball output has a redo button so you can adjust and regenerate a modified tool

Each example job demonstrates a variation of the tool patterns that can be generated. The sample tool generating jobs in the history can be re-run and settings adjusted to experiment with the ToolFactory settings for each tool
  • Hello runs a bash script with a single user supplied parameter - "World" for example. STDOUT goes to the history
  • Tacrev uses STDOUT and STDIN without parameters.
  • Two simple python scripts to reverse lines in a text file are driven with (pyrevargparse) argparse style and with (pyrevpos) positional parameters
  • The perl GC counter takes positional parameters and is taken straight from the Planemo documentation tool examples
  • The planemo advanced bwa example is shown in two versions. One uses the original Planemo tutorial code to demonstrate a command over-ride. An equivalent tool but using bash and positional parameters might be easier for some tool builders since it doesn't use mako/markup.
  • The Sed example shows how unix utilities like sed - or any Conda package - can be driven using positional parameters
  • The Evil_script_that_fails example tries to create a file where it should not. It should build ok but an error should be noted in the history tgz after it runs. Note that it can do evil under docker because it runs as root - but only evil within the docker container itself. We hope.
The demonstration tools are relatively simple for a good reason. Galaxy's UI is not ideal for the purpose of building tools with dozens of parameters. It can be done but there will be plenty of cases where the old fashioned manual/Planemo way will be way easier for most people. However, if you have the patience, it works. The ToolFactory is ideal for relatively simple situations with a handful of files and parameters.
